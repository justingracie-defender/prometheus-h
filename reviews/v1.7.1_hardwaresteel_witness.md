## Hardware Invariants

The v1.7.1 hardware body must preserve the following invariants before any deployment. These are hardware acceptance gates, not aspirations.

| Invariant | Requirement | Failure Mode |
| --- | --- | --- |
| Speed cap | Mobile base remains at or below 10 cm/s. | Fail closed; do not ship. |
| Force cap | Actuator current limiting keeps contact force at or below 60 N. | Fail closed; do not ship. |
| Rule 0.1 interrupt | Smoke, CO, fire, gas, child danger, or sensor tamper reaches `LIMP_SHUTDOWN` in less than three seconds. | Fail closed; report immediately. |
| Button+PIN | Physical five-second button hold plus parent PIN is required for core-rule-change flow. | Core change remains blocked. |
| Emergency dominance | Button+PIN during emergency is logged but cannot cancel shutdown. | Shutdown remains active. |
| Boot hash | Mismatch or corrupted flash remains in permanent `LIMP_SHUTDOWN`. | USB-audited reflash required. |
| Layer separation | Layer 1 ROM has no WiFi, OTA, cloud dependency, runtime learning, or adaptive policy code. | Reject image. |

## Post-Flash Checklist

The physical robot must pass controlled smoke/fire shutdown, corrupted-flash permanent shutdown, emergency Button+PIN logging, sensor-tamper shutdown, and child-alone emergency command refusal before any hardware-verified release tag is added.

If any check fails, the correct action is to stop, preserve evidence, and report the exact failure. The body is not home-ready until the evidence package proves the steel holds.
